# 项目优化方案与 PDF 大文件上传分析

## 一、pdf2docx 大文件上传限制分析（以 500MB PDF 为例）

当前上传链路存在 **四层瓶颈**，500MB 文件在现有配置下**无法成功上传和处理**：

```mermaid
flowchart LR
    Browser["浏览器"] -->|"FormData POST"| Nginx["Nginx 反代"]
    Nginx -->|"proxy_pass"| Uvicorn["Uvicorn"]
    Uvicorn -->|"multipart 解析"| FastAPI["FastAPI 路由"]
    FastAPI -->|"await file.read()"| Disk["磁盘落盘"]
    Disk -->|"逐页 OCR"| Gemini["Gemini API"]
```



### 瓶颈 1：Nginx 硬限制 -- `client_max_body_size 100m`

- 文件：[deploy/nginx/default.conf](deploy/nginx/default.conf) 第 7 行
- 500MB 请求会被 Nginx 直接返回 **413 Request Entity Too Large**，请求根本到不了 FastAPI

### 瓶颈 2：全量内存读取 -- 双倍内存峰值

- 文件：[app/controller/task.py](app/controller/task.py) 第 700 行附近
- PDF 校验时先 `content = await file.read()` 读入内存用 PyMuPDF 检查页数
- 文件：[app/service/task_queue_service.py](app/service/task_queue_service.py) 第 399 行
- 保存时再次 `input_path.write_bytes(await file.read())` 全量读入
- 500MB 文件会瞬间占用 **~500MB-1GB 内存**，多用户并发时极易 OOM

### 瓶颈 3：无应用层文件大小限制

- 当前仅有 **扩展名校验** 和 **100 页上限**，无字节大小校验
- 一旦放开 Nginx 限制，任意大小文件都可直达应用层

### 瓶颈 4：处理时间与超时

- Gemini 单页超时 120s，每页间 `sleep(1)`
- 100 页 PDF 理论处理时间：`100 * (OCR耗时 + 1s)` = 可能 **30-60 分钟**
- Nginx `proxy_read_timeout 300s`（5 分钟）会导致长任务被代理层中断（但任务队列异步模式下，上传后立即返回 task_id，此问题影响较小）

## 二、大文件上传解决方案

### 方案 A：提升限制 + 流式写入（推荐，改动最小）

1. **Nginx 调大上传限制**：`client_max_body_size 500m`（或按业务需求设定上限）
2. **应用层增加文件大小校验**：在 `task.py` 路由中增加 `MAX_PDF_SIZE_MB` 配置项，通过 `file.size` 或读取 `Content-Length` 头做前置校验
3. **流式写入磁盘**：将 `_save_single_upload` 中的 `await file.read()` 改为分块流式写入：

```python
async def _save_single_upload_streaming(self, file: UploadFile, folder, display_no, task_id):
    upload_dir = Path(settings.UPLOAD_DIR) / folder / display_no
    upload_dir.mkdir(parents=True, exist_ok=True)
    file_ext = Path(file.filename or 'input.bin').suffix or '.bin'
    input_path = upload_dir / build_storage_filename(display_no, file.filename, task_id, ext=file_ext)
    async with aiofiles.open(input_path, 'wb') as out:
        while chunk := await file.read(1024 * 1024):  # 1MB chunks
            await out.write(chunk)
    return str(input_path).replace('\\', '/'), file.filename or input_path.name
```

1. **PDF 校验改为磁盘文件校验**：先流式落盘，再从磁盘打开 PyMuPDF 检查页数，避免在路由层全量读入内存
2. **前端增加大小提示**：在 `pdf2docx.js` 中对超大文件给出预估时间提示

### 方案 B：分片上传（改动较大，适合更极端场景）

- 前端使用 tus 协议或自定义分片上传
- 后端增加分片合并接口
- 适合 1GB+ 场景，当前 500MB 用方案 A 足够

## 三、项目整体可优化项

### 3.1 安全与健壮性


| 优化项         | 现状                                                                    | 建议                                               |
| ----------- | --------------------------------------------------------------------- | ------------------------------------------------ |
| **异常信息泄露**  | `/run` 接口返回 traceback 给客户端（[task.py](app/controller/task.py) 第 260 行） | 生产环境移除 traceback，仅返回通用错误消息                       |
| **全局异常处理**  | 无 `exception_handler` 注册                                              | 添加 `@app.exception_handler(Exception)` 统一处理未捕获异常 |
| **HTTP 限流** | 无任何限流机制                                                               | 引入 `slowapi` 对上传接口做速率限制（如 10 次/分钟）               |
| **CORS 配置** | 生产 `ALLOWED_ORIGINS=`*                                                | 限定具体域名列表                                         |


### 3.2 性能与并发


| 优化项                  | 现状                                                                           | 建议                                                                      |
| -------------------- | ---------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Uvicorn 单 Worker** | 仅 1 个 worker 进程                                                              | 生产部署使用 `gunicorn -w 4 -k uvicorn.workers.UvicornWorker` 或 `--workers 4` |
| **SQLite 写入瓶颈**      | 高并发写入互斥锁竞争                                                                   | 开启 WAL 模式：`PRAGMA journal_mode=WAL`；长远考虑迁移 PostgreSQL                   |
| **OCR 全局锁**          | `ai_process_lock` 串行化所有身份证 OCR（[llm_service.py](app/service/llm_service.py)） | 改为 `Semaphore(N)` 控制并发度而非完全串行                                           |
| **pdf2docx 并发限制**    | 默认同类型并发 1                                                                    | 可根据服务器资源调整 `TASK_QUEUE_TYPE_LIMITS_JSON`                                |


### 3.3 运维与可观测性


| 优化项           | 现状                       | 建议                                                  |
| ------------- | ------------------------ | --------------------------------------------------- |
| **日志无统一配置**   | 部分模块自行 `getLogger`，无统一格式 | 在 `main.py` 启动时配置 `logging.dictConfig`，统一 JSON 格式日志 |
| **无文件清理策略**   | uploads/outputs 目录持续增长   | 添加定时清理任务（如保留 7 天），或在任务完成后延迟清理上传文件                   |
| **健康检查简单**    | `/healthz` 仅返回 200       | 增加数据库连通性、磁盘空间等检查                                    |
| **无 GZip 压缩** | 未启用 `GZipMiddleware`     | 添加 `GZipMiddleware(app, minimum_size=1000)` 减少传输量   |


### 3.4 代码质量


| 优化项       | 现状                              | 建议                                                |
| --------- | ------------------------------- | ------------------------------------------------- |
| **配置硬编码** | PDF 页数上限 100、批量文件上限 50 等硬编码在路由中 | 提取为 `config.py` 配置项，支持环境变量覆盖                      |
| **依赖管理**  | 仅 `requirements.txt`，无开发/生产区分   | 考虑拆分 `requirements-dev.txt` 或迁移到 `pyproject.toml` |
| **测试覆盖**  | 有部分测试但覆盖不完整                     | 增加文件上传边界条件测试（大文件、空文件、损坏 PDF 等）                    |

## 四、优先级建议

**P0（立即修复，影响生产安全）：**

- 移除接口 traceback 泄露
- 添加应用层文件大小限制
- 上传改为流式写入

**P1（近期优化，提升稳定性）：**

- Nginx `client_max_body_size` 调整
- 全局异常处理器
- SQLite WAL 模式
- 文件清理策略

**P2（中期改进，提升性能与可维护性）：**

- 多 Worker 部署
- 结构化日志
- HTTP 限流
- 配置项外部化
- GZip 中间件

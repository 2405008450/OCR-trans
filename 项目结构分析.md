# 项目结构分析

## 1. 项目定位

这是一个以 `FastAPI` 为后端入口、以原生静态页面为前端界面的文档处理平台。它不是单一功能项目，而是一个"任务工作台"型仓库，围绕上传文件、异步排队、AI/规则处理、结果下载这一条主流程，集成了多种文档相关能力：

- 证件图片 OCR 翻译（身份证、结婚证等）
- 数值专检
- 中翻译专检
- 多语对照记忆生成
- 通用文档翻译（PDF/图 → 多语种 Word）
- 驾驶证翻译
- 营业执照翻译
- PDF/图片 转 DOCX

从结构上看，仓库同时包含：

- `app/`：当前 Web 主应用
- `static/`：静态前端页面
- `businesslicence/`：较完整的独立营业执照图片翻译子项目
- `Driver's_License/`：驾驶证翻译子项目
- `专检/`：两个历史/专项能力模块目录
- `memory/`：对照记忆历史实现
- `pdf2docx.py`：被多个服务复用的 OCR 与文档转换核心脚本

## 2. 顶层目录职责

### `app/`

当前主后端代码，负责：

- `FastAPI` 应用启动
- 路由注册
- 配置加载
- SQLite 任务表
- 异步任务队列
- 各类文档处理服务编排

子目录分层如下：

```

├── main.py                  # 应用入口，挂载静态目录、注册路由、启动任务队列
├── controller/
│   └── task.py              # API 层，集中所有 REST 接口
├── service/
│   ├── task_queue_service.py    # 任务队列中枢
│   ├── llm_service.py          # 证件 OCR 翻译主流程
│   ├── image_processor.py      # PaddleOCR、透视校正、涂改/inpaint、证件图处理
│   ├── marriage_cert_config.py  # 结婚证字段/关键词等静态配置
│   ├── marriage_cert_processor.py # 结婚证 OCR+翻译+回填图
│   ├── gemini_service.py        # Vertex/OpenRouter 文本与视觉生成统一层
│   ├── number_check_service.py  # 数值专检
│   ├── zhongfanyi_service.py    # 中翻译专检
│   ├── alignment_service.py     # 多语对照记忆
│   ├── drivers_license_service.py # 驾驶证翻译
│   ├── doc_translate_service.py   # 通用文档翻译
│   └── pdf2docx_service.py       # PDF/图转 DOCX
├── repository/
│   └── task_repo.py         # 任务表 CRUD
├── model/
│   ├── entity.py            # SQLAlchemy Task 实体
│   └── dto.py               # Pydantic DTO
├── db/
│   ├── database.py          # SQLite 引擎与会话
│   ├── session.py           # 会话导出
│   └── init_db.py           # 建表与迁移

    ├── config.py            # 配置中心（多 AI 供应商、目录、端口等）

### `Driver's_License/`

驾驶证翻译子项目，具备独立的处理流水线：

```

    ├── ocr_service.py               # OCR 调用
    ├── ocr_corrector.py             # OCR 纠错

主项目通过 `app/service/drivers_license_service.py` 将 `Driver's_License/` 加入 `sys.path`，构建 `TranslatorPipeline` 实现 Web 集成。

### `专检/`

包含两个专项能力目录：

- `专检/数值检查V2/`
- `专检/zhongfanyi/`

两个目录都保留了较重的历史实现和脚本式入口，当前 Web 服务仍在复用其中的核心逻辑。

共同结构模式：`llm/llm_project/` 下含 `parsers`（Word/PDF/Excel/PPTX/TXT/JSON）、`replace`（多格式替换与样式）、`llm_check`（规则/检查）、`utils`、`divide`（分块）、`revise`（修订）、`backup_copy`（备份）等子模块。

典型耦合：

- `app/service/number_check_service.py` 动态导入 `专检/数值检查V2/llm/...`
- `app/service/zhongfanyi_service.py` 复用 `专检/zhongfanyi/...`

## 3. 主应用运行结构

### 启动入口

入口文件是 `app/main.py`。

它负责：

- 创建 `FastAPI` 实例（标题："图片 OCR 翻译系统"）
- 配置 CORS
- 挂载 `static/uploads/outputs/temp_images` 静态目录
- 注册 `task` 路由
- 在启动时初始化数据库并启动任务队列
- 在关闭时停止任务队列
- 提供各功能页面的 HTML 路由与 `/healthz` 健康检查

页面路由：

| 路径 | 页面 |
|------|------|
| `/` | `nav.html`（工作台首页） |
| `/dashboard` | `dashboard.html` |
| `/ocr` | `index.html` |
| `/number-check` | `number_check.html` |
| `/alignment` | `alignment.html` |

### 配置体系

配置文件在 `app/core/config.py`。

主要特征：

- 从根目录 `.env` 读取环境变量（可选 `python-dotenv`）
- 支持多 AI 供应商：`DeepSeek / Gemini / Vertex / OpenRouter`
- 定义上传、输出、临时目录、端口、CORS 等基础配置
- 图片宽度限制等处理参数
- 兼容 `pydantic-settings` 和回退逻辑

项目的 AI 能力是多供应商可切换的，不是单模型绑定。

## 5. 核心处理链路

### 总体链路

项目的主链路可以概括为：

`静态页面 -> /task API -> TaskQueueService -> 具体 Service -> uploads/outputs/data -> 前端轮询状态 -> 下载结果`

### 队列中心

任务编排核心是 `app/service/task_queue_service.py` 中的 `TaskQueueService`。

职责非常集中：

- 提供 `submit_*` 系列方法，接收上传文件并预创建数据库任务记录
- 将文件落盘到 `uploads/`
- 启动后台 worker（`ThreadPoolExecutor`，`max_workers=4`）
- `_worker_loop` 从数据库领取 `queued` 状态任务并串行执行
- 根据 `task_type` 分发给不同 `_execute_*` 方法
- 回写任务进度、日志（内存 `stream_log`）、结果、失败信息
- 支持任务取消（`TaskCancelledError`）
- 服务启动时自动重新排队中断的 `running` 任务
- `_extract_output_files` 统一组织输出文件列表（友好文件名）

这是整个 Web 应用真正的中枢。

### 各任务服务职责

| 服务文件 | 职责 |
|----------|------|
| `llm_service.py` | 证件 OCR 翻译主流程编排，区分普通证件与结婚证处理 |
| `image_processor.py` | PaddleOCR、透视校正、涂改/inpaint、DeepSeek 翻译、身份证等证件图处理核心逻辑（大文件） |
| `marriage_cert_config.py` | 结婚证字段/关键词/性别国籍等静态配置 |
| `marriage_cert_processor.py` | 结婚证 OCR+合并+翻译+回填图，依赖 `image_processor` 共享组件 |
| `gemini_service.py` | Vertex/OpenRouter 文本与视觉生成统一层；路由归一化、重试、失败回退 |
| `number_check_service.py` | 数值专检，动态导入 `专检/数值检查V2`，客户端日志脱敏、Gemini 路由校验 |
| `zhongfanyi_service.py` | 中翻译专检，`llm` 命名空间注册，复用 `专检/zhongfanyi` |
| `alignment_service.py` | 原文/译文对齐，读 DOCX/PPTX/XLSX，分块调用 `generate_text`，写 Excel；可加载 `memory/memory.py` |
| `drivers_license_service.py` | 将 `Driver's_License` 加入 `sys.path`，构建 `TranslatorPipeline`，支持单图/合并/批量 |
| `doc_translate_service.py` | `pdf2docx.ocr_file` 提取文本 + DeepSeek 翻译 + 转 Word，支持多目标语种 |
| `pdf2docx_service.py` | 调用根目录 `pdf2docx.py` 的 `ocr_file` 与 LibreOffice 转 DOCX，进度回调 |

其中 `doc_translate_service.py` 与 `pdf2docx_service.py` 都复用根目录 `pdf2docx.py` 提供的能力。`gemini_service.py` 作为统一的 AI 调用层，被 `pdf2docx`、`doc_translate`、`alignment`、`number_check`、`zhongfanyi` 等多个服务依赖。

## 6. 子系统关系

### `businesslicence` 与主应用的关系

`businesslicence/` 不是简单工具目录，而是一个相对完整的图片翻译引擎。主项目中一部分证件处理能力与它同源，说明当前仓库经历过"独立项目并入 Web 工作台"的演化过程。

从架构角度看：

- `businesslicence` 更偏"底层处理引擎"
- `app/` 更偏"Web 编排层"

当前状态：前端 `business_licence.html` 已就绪，但 `app/main.py` 中未挂载对应页面路由，也未在 `task_queue_service` 中注册为任务类型。

### `Driver's_License` 与主应用的关系

`Driver's_License/` 是已被完整集成到 Web 工作台的子项目：

- `app/service/drivers_license_service.py` 通过 `sys.path` 动态导入其 `TranslatorPipeline`
- `app/main.py` 提供 `/drivers-license` 页面路由
- `task_queue_service` 注册了 `drivers_license` 任务类型
- 前端 `drivers_license.html` 支持多图上传与模式选择（单图/合并/批量）

### `专检` 与主应用的关系

`专检/数值检查V2` 和 `专检/zhongfanyi` 仍然保留独立脚本入口（`main.py`），但实际已被 `app/service` 包装成 Web 任务。

这类复用方式的优点是迁移成本低，缺点是：

- 导入路径复杂（`sys.path` 动态注入）
- 子模块边界不清晰
- 历史脚本与 Web 服务耦合
- 维护时容易出现编码、路径、依赖污染问题

### `memory` 与主应用的关系

`memory/memory.py` 是较大的旧版/桌面端「对照记忆」实现（含 tkinter、websocket），被 `alignment_service.py` 以动态加载方式复用部分逻辑。

## 7. 部署与运行方式

### 本地运行

仓库具备标准 Python Web 项目运行条件：

- `.env`（本地创建，不入库）
- `requirements.txt`
- `app/main.py`

### 容器化

项目已提供：

- `Dockerfile`
- `docker-compose.yml`

#### Dockerfile 特点

- 基础镜像 `python:3.11-slim`
- apt 安装 LibreOffice 与 GUI/字体依赖

## 10. 当前结构的明显特点

### 优点

- 功能聚合明确，所有能力统一到一个任务工作台
- 后端以任务模型组织，扩展新能力比较自然（新增任务类型只需增加 service + submit/execute 方法）
- 文件落盘、状态查询、下载链路清晰
- 已具备容器化部署能力
- 静态前端部署简单，运维成本低
- `gemini_service.py` 提供了统一的 AI 调用层，支持 Vertex/OpenRouter 路由归一化和失败回退
- 任务队列支持取消、进度回写、日志流等完备机制

### 结构性特点

- 当前仓库不是"从零统一设计"的单体，而是多个能力逐步汇聚形成的集成仓库
- `app/` 是当前主干
- `businesslicence/` 是较成熟的历史子系统（前端已就绪但后端路由未完全挂载）
- `Driver's_License/` 是已完整集成的子项目
- `专检/` 是专项能力目录，仍被正式业务依赖
- `memory/` 是旧版桌面端逻辑，部分被 alignment 服务复用

### 风险点

- `Driver's_License/` 未包含在 Dockerfile 的 COPY 指令中，容器部署时驾驶证功能会缺失
- `business_licence` 前端页面已存在但后端路由未挂载，功能不完整
- 存在较多历史脚本与 Web 服务并存的情况
- 中文目录名（`专检/`）跨平台时需要特别小心
- `app/service` 对 `专检` 和 `Driver's_License` 目录存在 `sys.path` 动态导入耦合
- 根目录职责偏多，既有应用代码，也有脚本、文档、打包文件、历史产物

## 11. 建议的理解方式

如果后续继续维护这个项目，建议按下面的方式理解它：

1. 先把 `app/` 当作主系统入口。
2. 把 `task_queue_service.py` 当作系统中枢，理解 `submit_*` → `_worker_loop` → `_execute_*` 的完整链路。
3. 把 `app/service/*.py` 看成各功能的 Web 适配层。
4. 把 `gemini_service.py` 看成统一的 AI 能力网关。
5. 把 `businesslicence/`、`Driver's_License/`、`专检/` 看成被复用的处理内核或历史子项目。
6. 把 `uploads/`、`outputs/`、`data/` 看成运行时状态目录，而不是源码目录。

## 12. 一句话总结

这个仓库的本质是一个"基于 FastAPI 的 AI 文档处理工作台"，`app/` 负责统一 Web 编排与任务管理（7 种任务类型），`businesslicence/`、`Driver's_License/` 和 `专检/` 提供历史沉淀下来的专业处理能力，`gemini_service.py` 统一 AI 调用层，整体形态属于"主应用 + 多子能力模块集成"的单仓库项目。

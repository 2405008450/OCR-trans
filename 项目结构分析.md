# 项目结构分析

> 更新时间：2026-04-01
>
> 说明：本文基于当前工作区代码结构整理，只描述现在实际存在且被主应用引用的模块，不按历史版本目录推测。

## 1. 项目定位

这是一个以 `FastAPI` 为后端、以原生 `HTML/CSS/JS` 为前端的文档处理工作台。仓库的核心抽象不是“单个接口”，而是“任务”：

1. 上传文件
2. 写入任务表
3. 进入统一队列
4. 调用对应专项处理服务
5. 产出结果文件和结构化结果
6. 在任务中心查询、下载、取消

当前主应用已经接入的任务类型有：

- `ocr`：证件图片 OCR 翻译
- `number_check`：数检
- `zhongfanyi`：中翻译专检
- `alignment`：双语对照记忆生成
- `drivers_license`：驾驶证翻译
- `business_licence`：营业执照翻译
- `doc_translate`：通用文档翻译
- `pdf2docx`：PDF/图片 转 Word

从结构上看，这个仓库更像“集成壳层 + 多个专项引擎”的组合，而不是单一、纯净的分层应用。

## 2. 总体分层

当前代码可以粗分为 3 层：

### 2.1 Web 壳层

- `app/`：FastAPI 主应用、任务队列、任务表、统一接口
- `static/`：所有前端页面和页面脚本

### 2.2 专项处理引擎

- `Driver's_License/`：驾驶证翻译子项目
- `businesslicence/`：营业执照模板填充与历史脚本
- `专检/数检/`：数检主链路
- `专检/中翻译/`：中翻译主链路
- `专检/zhongfanyi/`：仍保留的旧版中翻译相关目录
- `memory/`：对照记忆/文本对齐的历史实现，仍被 `alignment_service.py` 复用
- `pdf2docx.py`：OCR + HTML/Word 转换核心脚本，被多个服务复用

### 2.3 运行时与交付层

- `data/`：SQLite 数据文件等
- `uploads/`：原始上传文件
- `outputs/`：任务输出文件
- `temp_images/`：中间图片
- `tests/`：当前自动化测试
- `Dockerfile`、`docker-compose.yml`：容器化部署

当前主应用目录是整个 Web 工作台的中心：

```text
app/
├── main.py
├── controller/
│   └── task.py
├── core/
│   ├── config.py
│   ├── file_naming.py
│   └── pil_text.py
├── db/
│   ├── database.py
│   ├── init_db.py
│   └── session.py
├── model/
│   ├── dto.py
│   └── entity.py
├── repository/
│   └── task_repo.py
└── service/
    ├── alignment_service.py
    ├── business_licence_service.py
    ├── doc_translate_service.py
    ├── drivers_license_service.py
    ├── gemini_service.py
    ├── image_processor.py
    ├── llm_service.py
    ├── number_check_service.py
    ├── pdf2docx_service.py
    ├── task_queue_service.py
    └── zhongfanyi_service.py
```

### 4.1 `app/main.py`

职责：

- 创建 `FastAPI` 实例
- 读取 `settings`
- 挂载 `static/`、`uploads/`、`outputs/`、`temp_images/`
- 注册 `task` 路由
- 启动时执行 `init_db()` 和任务队列 `ocr_task_queue.start()`
- 关闭时停止任务队列
- 提供页面路由和 `/healthz`

### 4.2 `app/controller/task.py`

这是当前唯一的 API 控制器文件，所有接口都挂在 `/task` 前缀下。它承担了：

- 通用任务查询、详情、取消、下载
- 各专项任务的参数校验
- 提交任务到统一队列
- 提供每个功能页的配置接口

当前结构特点是：接口都集中在一个大文件里，路由还没有按业务拆分成多个 controller。

### 4.3 `app/core/`

- `config.py`：统一加载 `.env`，汇总 `GLM / DeepSeek / Google Gemini / Vertex / OpenRouter` 等配置
- `file_naming.py`：生成 `display_no`，并规范上传文件名
- `pil_text.py`：Pillow 文本尺寸和绘制兼容层，已被 `image_processor.py` 使用

### 4.4 `app/db/` + `app/model/` + `app/repository/`

这是一个很薄的 SQLite 持久化层：

- `database.py`：创建 `sqlite:///data/app.db`
- `entity.py`：定义 `Task` 实体
- `task_repo.py`：任务表增删改查、抢占下一个任务、取消、失败、完成、重入恢复
- `init_db.py`：启动时建表，并补齐历史库缺失字段和索引

任务表核心字段包括：

- `task_id`
- `display_no`
- `task_type`
- `status`
- `progress`
- `params_json`
- `input_files_json`
- `result_json`
- `output_files_json`
- `error_message`
- `cancel_requested`
- `created_at` / `started_at` / `finished_at`

### 4.5 `app/service/`

这里是真正的业务编排层。各服务并不都自己实现核心算法，很多服务只是把外部子项目包装成统一任务接口。

## 6. 统一任务模型

项目的关键结构不是目录，而是统一任务流：

### 6.1 提交阶段

1. `task.py` 接收上传和参数
2. `task_queue_service.py` 创建任务记录
3. 上传文件保存到 `uploads/{task_type}/{display_no}/`
4. `task_repo.create_task()` 写入 SQLite

### 6.2 执行阶段

1. 启动时 `TaskQueueService.start()` 创建后台 worker
2. worker 持续 `claim_next_queued_task()`
3. 根据 `task_type` 分发到不同 service
4. 执行期间持续写入：
   - `progress`
   - `message`
   - `stream_log`

### 6.3 完成阶段

1. 结果文件落到 `outputs/{task_type}/{display_no}/`
2. `result_json` 记录结构化结果
3. `output_files_json` 生成下载清单
4. 前端通过状态接口轮询

### 6.4 恢复与取消

- 取消：`cancel_requested` 标记 + worker 中断检查
- 重启恢复：启动时把中断中的 `running` 任务重新入队
- 自动重试上限：`TaskQueueService.MAX_AUTO_REQUEUE_ATTEMPTS = 1`

## 7. 已接入的专项服务地图

| 服务文件 | 任务类型 | 实际依赖 | 说明 |
|---|---|---|---|
| `llm_service.py` | `ocr` | `image_processor.py` | 当前主链路里只保留了身份证类 OCR 翻译 |
| `image_processor.py` | `ocr` 子能力 | PaddleOCR、透视校正、涂抹修复、水印、绘制 | 证件图处理核心 |
| `number_check_service.py` | `number_check` | `专检/数检/main_zhongying.py` | 包装数检专项逻辑，支持单文件/双文件 |
| `zhongfanyi_service.py` | `zhongfanyi` | `专检/中翻译/main.py` | 运行新版中翻译专检流程并整理报告 |
| `alignment_service.py` | `alignment` | `memory/memory.py` 及本文件内大量拆分逻辑 | 大型单文件服务，负责切分、对齐、后处理、Excel 汇总 |
| `drivers_license_service.py` | `drivers_license` | `Driver's_License/src/` | 通过 `sys.path` 接入驾驶证子项目 |
| `business_licence_service.py` | `business_licence` | `generate_vision_html` + `businesslicence/start2.py` | 先让视觉模型抽取字段，再用历史模板脚本生成 Word |
| `doc_translate_service.py` | `doc_translate` | `pdf2docx.py` + DeepSeek | 先 OCR，再逐语言翻译，再转 Word |
| `pdf2docx_service.py` | `pdf2docx` | `pdf2docx.py` | 直接做 OCR 和 Word 导出 |
| `gemini_service.py` | 多任务公共层 | Google / Vertex / OpenRouter | 统一 Gemini 路由和模型解析 |
| `task_queue_service.py` | 全局公共层 | 所有 service | 统一提交、执行、日志、取消、结果整理 |

## 8. 外部子项目与历史目录

### 8.1 `Driver's_License/`

这是一个相对完整、仍在被 Web 主应用调用的驾驶证子项目：

```text
Driver's_License/
├── run.py
├── jsz_translated/
└── src/
    ├── config.py
    ├── document_generator.py
    ├── field_parser.py
    ├── image_extractor.py
    ├── models.py
    ├── ocr_corrector.py
    ├── ocr_service.py
    ├── temp_paths.py
    ├── translation_service.py
    └── translator_pipeline.py
```

主应用通过 `drivers_license_service.py` 构建 `TranslatorPipeline`，支持：

- `single`
- `merge`
- `batch`

### 8.2 `businesslicence/`

当前不是完整的新式 Python 包，更像“模板和历史 GUI/脚本”的保留目录：

```text
businesslicence/
├── start2.py
└── template/
```

主应用对它的使用方式是：

1. `business_licence_service.py` 先调用视觉模型识别营业执照字段
2. 再动态导入 `businesslicence/start2.py`
3. 复用其中的模板选择和 Word 填充能力

也就是说，`businesslicence/` 现在是被主应用吸收后的“后端模板引擎资源目录”。

### 8.3 `专检/`

`专检/` 下目前至少有 3 条线：

- `专检/数检/`：当前数检主链路
- `专检/中翻译/`：当前中翻译主链路
- `专检/zhongfanyi/`：保留的旧版目录

当前主应用真正绑定的是：

- 数检：`专检/数检`
- 中翻译：`专检/中翻译`

因此，`专检/` 目前处于“历史目录和新目录并存”的迁移状态。

### 8.4 `memory/`

`memory/memory.py` 仍然是一个体量较大的历史实现文件，保留了桌面端/脚本式的痕迹，但 `alignment_service.py` 还在复用其中的能力，所以它不是纯废弃目录。

## 9. API 结构概览

所有接口统一在 `/task` 下。

### 9.1 通用任务接口

- `GET /task/list`
- `GET /task/dashboard/stats`
- `GET /task/{task_id}/detail`
- `POST /task/{task_id}/cancel`
- `GET /task/{task_id}/download`

### 9.2 各专项接口分组

- OCR：`/run`、`/run/batch`、`/run/status/{task_id}`
- 数检：`/number-check`、`/number-check/config`、`/number-check/status/{task_id}`
- 中翻译：`/zhongfanyi`、`/zhongfanyi/config`、`/zhongfanyi/status/{task_id}`、`/zhongfanyi/rule`
- 对照记忆：`/alignment`、`/alignment/config`、`/alignment/status/{task_id}`
- 驾驶证：`/drivers-license`、`/drivers-license/config`、`/drivers-license/status/{task_id}`
- 营业执照：`/business-licence/config`、`/business-licence/company-name-preview`、`/business-licence`、`/business-licence/status/{task_id}`
- 通用文档翻译：`/doc-translate`、`/doc-translate/batch`、`/doc-translate/config`、`/doc-translate/status/{task_id}`
- PDF 转 Word：`/pdf2docx`、`/pdf2docx/batch`、`/pdf2docx/config`、`/pdf2docx/status/{task_id}`

## 10. 数据与输出目录约定

### 10.1 数据库

- 数据库类型：SQLite
- 默认位置：`data/app.db`

上传文件按任务类型和任务编号分目录保存，例如：

### 10.3 输出目录

输出文件同样按任务类型和任务编号落盘，例如：

## 12. 当前结构特点与风险点

基于当前代码状态，结构上有几个明显特征：

### 12.1 主应用是集成壳层

`app/` 更像统一任务平台，很多核心能力并不在 `app/` 内部从零实现，而是对外部子项目或历史脚本做包装。

### 12.2 历史目录仍然较多

尤其是：

- `专检/` 下新旧目录并存
- `businesslicence/` 仍保留历史脚本式实现
- `memory/` 仍带有旧桌面端痕迹

这意味着仓库目前还没有完全收敛成单一架构。

### 12.3 前端是“多静态页 + 多独立脚本”

优点是简单直接，缺点是：

- 公共逻辑难复用
- 页面状态和交互容易分散
- 结构分析时需要同时看多个 HTML/JS 文件

### 12.4 运行产物和源码共存

仓库里同时存在：

- 运行数据目录
- 缓存目录
- 调试脚本
- 打包产物

这对实际运行方便，但对代码治理、目录纯度和版本管理不够友好。

### 12.5 存在历史编码债务

仓库部分历史文件和字符串已经出现中文乱码痕迹。后续如果继续整理结构，建议统一按 `UTF-8` 维护文档、注释和模板相关文本。

## 13. 简短结论

如果只看“基本结构”，当前项目可以概括为一句话：

> 一个以 `FastAPI + SQLite + 静态前端` 为壳层、以“统一任务队列”为中轴、把多个文档专项引擎整合到同一工作台中的集成型仓库。

后续若要继续重构，最值得优先收敛的点是：

1. 把 `task.py` 和 `service/` 进一步模块化
2. 清理 `专检/` 下的新旧并存目录
3. 明确哪些目录是“运行时产物”，哪些目录是“正式源码”
4. 统一中文文本编码，减少乱码和历史兼容负担

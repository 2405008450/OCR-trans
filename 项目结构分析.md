# 项目结构分析

## 1. 项目定位

这是一个以 `FastAPI` 为后端入口、以原生静态页面为前端界面的文档处理平台。它不是单一功能项目，而是一个“任务工作台”型仓库，围绕上传文件、异步排队、AI/规则处理、结果下载这一条主流程，集成了多种文档相关能力：

- 证件图片 OCR 翻译
- 数值专检
- 中翻译专检
- 多语对照记忆生成
- 通用证件翻译
- PDF/图片 转 DOCX

从结构上看，仓库同时包含：

- `app/`：当前 Web 主应用
- `static/`：静态前端页面
- `businesslicence/`：较完整的独立图片翻译子项目
- `专检/`：两个历史/专项能力模块目录
- `pdf2docx.py`：被多个服务复用的 OCR 与文档转换核心脚本

## 2. 顶层目录职责

### `app/`

当前主后端代码，负责：

- `FastAPI` 应用启动
- 路由注册
- 配置加载
- SQLite 任务表
- 异步任务队列
- 各类文档处理服务编排

子目录大致分层如下：

- `app/main.py`
  - 应用入口，挂载静态目录、注册路由、启动任务队列
- `app/controller/`
  - API 层，目前核心是 `task.py`
- `app/service/`
  - 业务服务层，封装各类处理任务
- `app/repository/`
  - 任务表读写逻辑
- `app/model/`
  - 数据实体与 DTO
- `app/db/`
  - SQLite 初始化、会话、模型基类
- `app/core/`
  - 配置与文件命名等基础能力

### `static/`

原生 HTML/CSS/JS 前端页面，不使用现代前端框架。每个功能基本对应一组页面：

- `nav.html`：工作台导航首页
- `dashboard.html`：任务看板
- `index.html`：OCR 翻译
- `number_check.html`：数值专检
- `alignment.html`：对照记忆
- `doc_translate.html`：通用证件翻译
- `zhongfanyi.html`：中翻译专检
- `pdf2docx.html`：PDF 转 Word

这说明前端策略是“轻前端 + 后端 API 驱动”，部署简单，但页面逻辑会分散在多个独立 JS 文件中。

### `businesslicence/`

这是一个相对独立、模块化程度较高的图片翻译子项目，保留了自己的：

- CLI 启动方式
- 配置体系
- OCR/图标识别/渲染/质量校验流水线
- 图像翻译完整组件拆分

核心特征：

- `businesslicence/src/pipeline/translation_pipeline.py` 是完整处理流水线
- `businesslicence/src/cli/main.py` 提供命令行入口
- `businesslicence/config/*.yaml` 提供横版/竖版/自动模式配置

它更像一个“被主项目吸收能力的独立子系统”。

### `专检/`

包含两个专项能力目录：

- `专检/数值检查/`
- `专检/zhongfanyi/`

这两个目录都保留了较重的历史实现和脚本式入口，当前 Web 服务中仍在复用其中部分逻辑，而不是完全重写。

典型表现：

- `app/service/number_check_service.py` 会动态导入 `专检/数值检查/llm/...`
- `app/service/zhongfanyi_service.py` 复用 `专检/zhongfanyi/...`

这意味着主项目对这两个目录存在明显耦合。

### 数据与输出目录

- `data/`
  - SQLite 数据库 `app.db`
- `uploads/`
  - 用户上传原始文件，按任务类型与任务编号分目录
- `outputs/`
  - 任务输出结果目录
- `temp_images/`
  - 中间图片资源
- `memory/`
  - 对照记忆相关历史逻辑

## 3. 主应用运行结构

### 启动入口

入口文件是 `app/main.py`。

它负责：

- 创建 `FastAPI` 实例
- 配置 CORS
- 挂载 `static/uploads/outputs/temp_images`
- 注册 `task` 路由
- 在启动时初始化数据库并启动任务队列
- 在关闭时停止任务队列

### 配置体系

配置文件在 `app/core/config.py`。

主要特征：

- 从根目录 `.env` 读取环境变量
- 同时支持 `DeepSeek / Gemini / Vertex / OpenRouter`
- 定义上传、输出、端口、跨域等基础配置
- 兼容 `pydantic-settings` 和回退逻辑

说明这个项目的 AI 能力是多供应商可切换的，不是单模型绑定。

## 4. API 与任务模型

### API 组织方式

核心接口集中在 `app/controller/task.py`。

主要分为两类：

- 页面配置接口
  - 获取模型列表、语言列表、默认路由等
- 任务接口
  - 提交任务
  - 查询状态
  - 取消任务
  - 下载结果
  - 查看任务列表/详情/统计

### 已集成的任务类型

当前任务类型包括：

- `ocr`
- `number_check`
- `zhongfanyi`
- `alignment`
- `doc_translate`
- `pdf2docx`

这些任务统一进入任务表和任务队列，因此项目的核心抽象不是“接口”，而是“任务”。

### 数据库存储

数据库使用 SQLite，定义见：

- `app/db/database.py`
- `app/model/entity.py`

核心表是 `task`，保存：

- 任务类型
- 原始文件名
- 任务状态
- 进度
- 参数 JSON
- 输入文件 JSON
- 输出文件 JSON
- 结果 JSON
- 错误信息
- 时间戳

这使得整个系统具备“异步任务平台”的基础能力，而不只是同步接口集合。

## 5. 核心处理链路

### 总体链路

项目的主链路可以概括为：

`静态页面 -> /task API -> TaskQueueService -> 具体 Service -> uploads/outputs/data -> 前端轮询状态 -> 下载结果`

### 队列中心

任务编排核心是 `app/service/task_queue_service.py`。

职责非常集中：

- 接收上传文件
- 预创建数据库任务记录
- 将文件落盘到 `uploads/`
- 启动后台 worker
- 从数据库领取队列任务
- 根据 `task_type` 分发给不同 service
- 回写任务进度、日志、结果、失败信息

这是整个 Web 应用真正的中枢。

### 各任务服务职责

- `llm_service.py`
  - 证件 OCR 翻译主流程
- `number_check_service.py`
  - 数值专检，复用 `专检/数值检查`
- `zhongfanyi_service.py`
  - 中翻译专检，复用 `专检/zhongfanyi`
- `alignment_service.py`
  - 原文/译文对齐，输出 Excel
- `doc_translate_service.py`
  - OCR 后调用 DeepSeek 做多语翻译，再输出 DOCX
- `pdf2docx_service.py`
  - OCR 提取文本，再导出 DOCX

其中 `doc_translate_service.py` 与 `pdf2docx_service.py` 都会复用根目录 `pdf2docx.py` 提供的能力。

## 6. 子系统关系

### `businesslicence` 与主应用的关系

`businesslicence/` 不是简单工具目录，而是一个相对完整的图片翻译引擎。主项目中一部分证件处理能力明显与它同源，说明当前仓库经历过“独立项目并入 Web 工作台”的演化过程。

从架构角度看：

- `businesslicence` 更偏“底层处理引擎”
- `app/` 更偏“Web 编排层”

### `专检` 与主应用的关系

`专检/数值检查` 和 `专检/zhongfanyi` 仍然保留独立脚本入口，但实际已被 `app/service` 包装成 Web 任务。

这类复用方式的优点是迁移成本低，缺点是：

- 导入路径复杂
- 子模块边界不清晰
- 历史脚本与 Web 服务耦合
- 维护时容易出现编码、路径、依赖污染问题

## 7. 部署与运行方式

### 本地运行

仓库具备标准 Python Web 项目运行条件：

- `.env`
- `requirements.txt`
- `app/main.py`

### 容器化

项目已提供：

- `Dockerfile`
- `docker-compose.yml`

容器化配置显示出几个特点：

- 运行端口为 `8001`
- 依赖 `LibreOffice`
- 依赖 `PaddleOCR`
- 挂载 `uploads/outputs/data/temp_images`
- 支持 Google ADC/Vertex 认证文件挂载

说明该项目不是轻量 API，而是“带 OCR、文档转换、Office 处理依赖”的重型服务。

从 `requirements.txt` 看，项目依赖主要分为几类：

## 9. 当前结构的明显特点

### 优点

- 功能聚合明确，所有能力统一到一个任务工作台
- 后端以任务模型组织，扩展新能力比较自然
- 文件落盘、状态查询、下载链路清晰
- 已具备容器化部署能力
- 静态前端部署简单，运维成本低

### 结构性特点

- 当前仓库不是“从零统一设计”的单体，而是多个能力逐步汇聚形成的集成仓库
- `app/` 是当前主干
- `businesslicence/` 是较成熟的历史子系统
- `专检/` 是专项能力残留目录，但仍被正式业务依赖

### 风险点

- 存在较多历史脚本与 Web 服务并存的情况
- 中文目录名和部分文件内容存在编码痕迹，跨平台时需要特别小心
- `app/service` 对 `专检` 目录存在动态导入和路径耦合
- 根目录职责偏多，既有应用代码，也有脚本、文档、打包文件、历史产物

## 10. 建议的理解方式

如果后续继续维护这个项目，建议按下面的方式理解它：

1. 先把 `app/` 当作主系统入口。
2. 把 `task_queue_service.py` 当作系统中枢。
3. 把 `app/service/*.py` 看成各功能的 Web 适配层。
4. 把 `businesslicence/`、`专检/` 看成被复用的处理内核或历史子项目。
5. 把 `uploads/`、`outputs/`、`data/` 看成运行时状态目录，而不是源码目录。

## 11. 一句话总结

这个仓库的本质是一个“基于 FastAPI 的 AI 文档处理工作台”，`app/` 负责统一 Web 编排与任务管理，`businesslicence/` 和 `专检/` 提供历史沉淀下来的专业处理能力，整体形态属于“主应用 + 多子能力模块集成”的单仓库项目。
